# ROADMAP

> This is what we are concretely building right now.
> See `vision.md` for why.

---

## Phase 1 — MVP (Closed Beta)

**Goal:** A non-technical user signs in, uses a single Hermes agent or a team of Hermes agents with defined roles, and can accomplish real goals — no setup, no CLI, no API keys.

**Success looks like:** The builder can use it daily for side projects and show it to friends and family to get meaningful feedback.

---

### 1. Infrastructure — Multi-Tenancy Layer

Hermes was built as a single-user local process. We are making it multi-tenant on the cloud without forking the library heavily.

**Stack:**
- **Neon DB (Postgres)** — primary database for user data, agent metadata, and secrets
- **GCP Compute Engine** — runs Firecracker microVMs for agents
- **GCP Cloud Run** — hosts the combined backend and frontend container
- **Google Cloud Volumes** — persists agent memory and state across sessions
- **Firebase** — authentication (Google OAuth) and real-time updates

**Multi-tenancy approach:**
- One Firecracker microVM per agent (not per user). Each agent runs in its own isolated VM.
- User onboarding provisions one or more VMs scoped to that user — one per role if they choose a team.
- Agent VMs are stateful: each has its own Hermes memory store persisted in Google Cloud Volumes, private per agent.
- Agent lifecycle management: start, stop, restart VMs without losing memory or context.

*Why Firecracker:* strong isolation boundary per agent, fast boot, low overhead — lets us run many agents without the cost of full VMs.

**Hermes wrapper:**
We are building a wrapper around the Hermes library rather than forking it. The wrapper must support all services Hermes depends on — multiple LLM providers, browser use, code execution, Honcho memory, and any other integrations Hermes uses — so that hosted agents have full feature parity with self-hosted Hermes.

### 2. Agent Teams — Roles, Communication, Shared Knowledge

A user gets either a single agent or a coordinated team with defined roles.

- **Roles:** each agent in the team has a role (e.g. Researcher, Executor, Planner, Domain Expert). Roles define the agent's system prompt, tools available, and scope of responsibility.
- **Inter-agent chat:** agents can message each other to delegate, report back, or ask questions. This uses Hermes's existing multi-platform gateway adapted for internal A2A messaging.
- **Shared team folder:** a common knowledge directory all agents in the team can read from and write to. This is where shared context, generated knowledge, and task artifacts live. Individual agent memory stays private to that agent.
- Default team composition for MVP: a sensible starting set of 3–5 roles that cover most user goals without overwhelming them.

---

### 3. GitHub Integration

Agents that generate code need somewhere to put it. Rather than storing code only in the shared team folder, we integrate with GitHub so users own and can access their generated code outside our platform.

- User connects their GitHub account during or after onboarding.
- Code generated by agents is committed to a user-owned repo.
- This aligns with the vision: user owns the output, not just us.

---

### 4. Onboarding — Zero Setup

The entire value prop of Phase 1 over raw Hermes is that non-technical users can actually use it.

- User signs in with Google OAuth.
- On first login, a default agent is provisioned automatically in the background.
- User lands in a web UI with their agent ready — no configuration required.
- Managed LLM credentials: we provide the API keys, users do not need their own.
- User can optionally create a team, select roles, or adjust defaults — but none of this is required to get started.

---

### 5. Web App UI

Hermes is CLI-only today. We need a web interface that makes agents accessible to non-technical users.

- Chat interface per agent — talk to individual agents or the whole team.
- Visibility into what agents are doing (tool calls, browsing, writing files) — users should not feel like things are happening in a black box.
- Shared team folder visible in the UI — users can see what knowledge the team has built.
- Mobile-friendly — beta users will want to check in on agents from their phones.

---

### 6. Security Hardening

Hermes passes raw context to LLMs. This must be addressed before any users outside the internal team are invited.

- Secrets and credentials typed by users must never be forwarded to the LLM in plain text.
- User-provided credentials (passwords, API keys) are stored encrypted in Neon DB — keeping secrets in our primary DB avoids adding another vendor dependency.
- Secrets are injected as environment variables into agent VMs via placeholders — they are never present in the VM's conversation context.
- Audit what Hermes currently sends to the LLM and identify any other sensitive surface areas.

---

### Phase 1 Exit Criteria

- [ ] User can sign in and have an agent ready with zero manual setup
- [ ] User can optionally create a team with distinct roles
- [ ] Agents can communicate with each other and use the shared team folder
- [ ] No user secrets are passed raw to the LLM
- [ ] Web UI is usable on mobile
- [ ] GitHub integration working — generated code committed to user-owned repo
- [ ] At least one complete end-to-end workflow demonstrated (e.g. research a topic → produce a document → store in team folder → commit code to GitHub)

---

*Phase 2 (A2A between users, P2P network) is not scoped until Phase 1 exit criteria are met.*
